The client for this project was The Gaming Room, a company that wanted to expand its Android game, Draw It or Lose It, into a web-based application that could run on multiple operating systems. The application needed to support multiple games, teams, and players while ensuring that game and team names remained unique. It also needed to use a client-server architecture so that users on Windows, macOS, Linux, Android, and iOS could all access the same application through a web browser.

I think I did a good job organizing the software design document and explaining my design decisions clearly. I described the software requirements, evaluated different operating platforms, and provided recommendations that connected back to the client's needs. I also explained how design patterns such as the Singleton and Iterator patterns would help meet the project's requirements.

Creating the design document before writing code made the implementation process much easier. The document helped me understand the relationships between the different classes and how they would interact with one another. Having the UML diagram and requirements already planned made it easier to implement the Java classes and understand why design patterns were being used.

If I could revise one part of the project, I would spend more time expanding the platform evaluation and recommendations. While I believe they addressed the client's needs, I could provide more detailed comparisons between operating systems and discuss additional cloud hosting options and deployment strategies. This would make the recommendations even stronger for a real client.

I focused on the client's main goals of expanding the game to multiple platforms while maintaining performance, security, and scalability. I selected a client-server architecture, recommended Linux for hosting, and incorporated design patterns that ensured unique game data and efficient resource management. Considering the user's needs is important because software should solve the client's problem rather than simply demonstrate technical skills. Designing around the client's requirements helps create software that is practical, reliable, and easier to maintain.

I approached the design by first identifying the client's requirements and then selecting the architecture and design patterns that best met those requirements. Breaking the project into smaller pieces made the overall design easier to understand and implement. In the future, I would continue using UML diagrams, design documents, and incremental development to plan projects before writing code. I would also spend more time considering scalability, security, and future expansion during the design phase since these factors become increasingly important as software grows.
